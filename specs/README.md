# Specs

Framework-agnostic source of truth for the ComPsych Design System.

## What lives here

- **`foundations.md`** — the foundations spec: color, typography, spacing, elevation, icons, radius, border width, and the four-tier theming model. Human-readable companion to the token JSON.
- **`_template.spec.md`** — the canonical component spec template. Every component spec is created by copying this file.
- **`<component-name>.spec.md`** — one file per component. Kebab-case filename, PascalCase heading.

## How specs are used

Specs are the authoritative definition of each component. They describe:

- **Purpose** — what the component is for.
- **API contract** — props, events, slots, with web/RN naming differences.
- **Visual specification** — which `sys.*` tokens drive each visual property.
- **States** — default, hover, focus, pressed, selected, disabled, loading, error.
- **Behavior** — interactions, edge cases, transitions.
- **Accessibility** — per-framework (web and React Native).
- **Reference implementation paths** — where the React reference lives.

Angular and React Native implementations are generated by Claude Code sessions inside ComPsych's own project repos, using the adopter `CLAUDE.md` files and these spec files as input.

## The critical rule

Every `sys.*` token referenced in a Visual specification must be a `sys.*` token. Never reference `core.*`, `product.*`, or `brand.*`. Never inline raw hex codes, raw px values, or raw font sizes.

## Adding a new component spec

1. Copy `_template.spec.md` to `<component-name>.spec.md`.
2. Fill in every section. Delete the HTML comment guidance in the template before publishing.
3. Add a manifest entry in the repo root `manifest.json` under `components`.
4. Once the spec reaches `stable`, implement in the React reference app. Adopters can then generate Angular and RN implementations from the spec.

## Status values

- `draft` — in progress, not ready for consumers.
- `review` — pending review by Konpo + ComPsych stakeholders.
- `stable` — approved, implemented in reference, ready for adopters to generate framework-specific implementations.
